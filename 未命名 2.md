# 【开发异常日志解读】大模型评测计划书

## 一、评测模型列表

本次评测选取国内三大开源模型系列（DeepSeek、Qwen、GLM）中最新、最具代表性的5个模型，覆盖从轻量到重型的不同规模，并特别关注其架构差异（MoE与Dense）带来的能力区别。

### 1.1 国内开源模型

|序号|模型名称|所属机构|模型特点（评测关注点）|
|---|---|---|---|
|1|**DeepSeek-R1**​|深度求索|满血版671B MoE模型，代表当前开源模型的顶级复杂推理与代码分析能力上限。|
|2|**DeepSeek-V3.2**​|深度求索|平衡版MoE模型，在推理能力、输出长度与成本效率间取得较优平衡，适合日常高频使用。|
|3|**Qwen2.5-72B**​|阿里|大规模Dense模型，在推理、指令遵循及智能体能力方面表现全面，是综合能力标杆。|
|4|**Qwen3-14B**​|阿里|高效Dense模型，在中等参数量下提供优秀的推理与指令遵循能力，是成本效益的考察重点。|
|5|**GLM-4.7**​|智谱|具备更强编程与多步骤推理/执行能力的专项优化模型，用于测试其在日志分析场景的专业性。|

## 二、评测目标

本次评测旨在深入考察国内顶尖开源大模型在"工程化日志分析管道"中的综合表现，重点评估其多步骤推理、智能体协作潜力、不同架构（MoE vs Dense）的效率差异，以及面向真实项目（STARCLAW）的实用性与部署成本效益，最终为AI在运维领域的工程化落地提供选型指南。

具体目标包括：

1. 对比不同架构（MoE vs Dense）在处理复杂日志分析任务时的性能表现
    
2. 评估从14B到671B不同规模模型的能力提升曲线与成本效益
    
3. 测试模型在真实工程场景（STARCLAW项目）中的实际应用价值
    
4. 探索端云协同架构下，不同模型的部署策略与成本优化方案
    

## 三、评测内容设计

评测将模拟真实运维SRE工作流，从简单到复杂分为四个层次：

### 3.1 工程化预处理管道对比测试

- **目的**：验证"预处理浓缩日志"对提升大模型分析效果、降低成本的效用
    
- **方法**：对同一批原始日志，分别采用"原始日志直接分析"和"预处理（关键词过滤、聚类）后分析"两种方式，对比各模型在效果与成本上的差异
    
- **测试数据**：包含10个不同类型的日志文件，总计约50,000行日志
    
- **评估指标**：问题定位准确率、根因分析深度、Token消耗、响应时间
    

### 3.2 基础能力测试

- **目的**：评估模型对单条错误日志的基础理解能力
    
- **测试用例**：包含20个常见运行时错误（如`NullPointerException`）、系统错误（如`Connection refused`）、资源错误（如`OutOfMemoryError`）等
    
- **考察点**：
    
    - 错误类型识别准确性
        
    - 根因分析合理性
        
    - 修复建议可操作性
        
    - 输出结构化程度
        
    

### 3.3 复杂场景测试

- **目的**：考察模型在多条关联日志、跨服务链路场景下的逻辑推理与问题定位能力
    
- **测试用例**：包含5个复杂场景：
    
    1. 微服务调用链异常（3个服务，15条关联日志）
        
    2. Kubernetes Pod启动故障（包含镜像拉取、资源调度、健康检查等多阶段日志）
        
    3. 分布式系统并发问题（涉及锁竞争、事务回滚）
        
    4. 数据库连接池泄露（慢查询、连接数异常增长）
        
    5. 缓存雪崩场景（Redis集群故障引发的连锁反应）
        
    
- **考察点**：关联分析能力、根因定位准确性、推理步骤的清晰度、自我纠正能力
    

### 3.4 专项对比

- **架构对比**：MoE架构 (DeepSeek) 与 Dense架构 (Qwen, GLM) 在相同任务下的效果、速度与成本差异
    
- **规模对比**：从14B到671B，参数量增长带来的能力提升曲线与成本代价
    
- **端云部署考量**：评估Qwen3-14B、DeepSeek-V3.2等模型作为边缘侧本地化部署候选的可行性，包括资源消耗、响应速度、分析质量
    
- **多轮对话能力**：测试模型在迭代提问、持续交互场景下的表现

## 五、评测方法论

### 5.1 测试环境

- **API平台**：各模型官方平台（DeepSeek、阿里云灵积、智谱AI开放平台）
    
- **测试工具**：基于Python开发自动化测试框架，支持并发测试、结果记录、性能监控
    
- **数据存储**：使用SQLite存储测试结果，JSON格式存储原始日志和模型输出
    

### 5.2 评分体系

采用百分制加权评分，各项权重如下：

|评分维度|权重|评估标准|
|---|---|---|
|**准确性**​|40%|根因识别正确率、错误分类准确度|
|**实用性**​|25%|修复建议可操作性、输出结构化程度|
|**效率**​|20%|响应时间、Token使用效率|
|**深度**​|15%|分析维度全面性、多步推理能力|

### 5.3 成本核算方法

- **API成本**：记录每次调用的输入输出Token数，按各平台公开单价计算
    
- **时间成本**：记录端到端响应时间，折算人力成本
    
- **综合性价比**：评分总分 / （API成本 + 时间成本）

## 八、风险与应对

### 8.1 技术风险

|风险描述|可能性|影响程度|应对措施|
